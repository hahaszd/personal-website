# 个人网站 · 建设方案与构想（交接文档）

> 这是一份"单一来源"交接文档。把它拷贝到新的网站项目文件夹后，新窗口的 AI agent 读这一份就能拿到全部上下文：**为什么做（初衷）、做成什么样（设计原则）、怎么做（实现方案）、先做什么（执行步骤）**。
>
> 给新窗口 agent 的话：请先用 **Plan 模式**通读本文件并出一份落地计划，与我确认后再执行。不要把这个站做成"普通博客"——它在我整盘里是有特定使命的"地基"，见下。

---

## Part A · 初衷与构想（最重要，先读）

### 一句话定位
我的个人网站 = 我在 AI 时代的**「地基」**：一个自己掌控、开放、可被通用 AI 检索与引用的主场，用来沉淀我的长文与项目实践记录。

### 为什么做（底层逻辑）
- 我在围绕「**AI 时代的个人品牌 / reputation**」持续输出，自我定位是「**AI 时代的向导 / 翻译者**」——把复杂的 AI 讲清楚、带更多人跨过"不知道 AI 能做什么"这道认知门槛。
- X、小红书、公众号都是**平台**，受算法和封闭生态制约，且站外通用 AI 往往抓不到（公众号尤其封闭）。我需要一个**不依赖平台、可追溯、长期复利**的资产。
- reputation 有一个**无法作弊的维度是「时间」**——自有域名上公开、有日期、可检索的记录，正是用来"长"出这种信任的载体。
- 纪律：**科普是放大器，不是护城河**；讲"是什么"的同时必须挂靠在真实实践上（"我真做了什么、踩了什么坑、我的判断"）。所以网站不仅要有文章，还要有**项目实践记录**。

### 给谁看（双重受众，同等重要，缺一不可）
1. **人**：对 AI、个人品牌、用 AI 做事感兴趣的读者，以及潜在的合作方 / 客户。
2. **通用 AI**：ChatGPT / Claude / Perplexity 等。我希望它们能**抓取、理解、并在相关话题或被问到我本人时引用**这里的内容。这条受众决定了很多技术取舍（见 Part B）。

### 网站要承载什么
- **博客**：长文 / 感悟 / 实践记录 / AI 使用心得。内容源来自我的写作仓库的成稿（见 Part C）。
- **项目展示**：我正在做 / 做过的项目，按「做了什么、怎么做、踩了什么坑、结果、日期」的**实践记录**方式呈现，链接到 GitHub（我已有 2 个公开项目）。
- **关于我**：一句话说清「我是谁、在做什么」，承接我在 X 上的定位。

### 调性
**克制、深度、真实**。视觉克制、排版利落、以内容为主；语气像"我"本人，不堆砌、不标题党、不做流量噱头。当"涨粉/流量"打法与"克制深度"取向冲突时，以克制深度为准。

## Part B · 设计原则（据此做技术与页面决策）

1. **为「被 AI 引用」而优化**（最高优先级之一，区别于普通博客）：
   - 干净的语义化 HTML、静态可直出（SSG）。
   - 完整 metadata / OpenGraph / Twitter Card，每篇文章有标题、描述、日期、canonical。
   - 输出 `sitemap.xml`、`RSS`，并考虑 `llms.txt`（专门给 LLM 的内容索引）。
   - 内容以纯文本/HTML 形式可被爬虫直接读到，不要依赖客户端 JS 才能渲染正文。
2. **双语就绪**：先做中文，但**路由 / 架构要预留中英双语**（英文内容被 AI 引用概率更高）。不要写死成单语，免得以后重构。理想是同一篇可挂中/英两版。
3. **克制、深度、真实**：见 Part A 调性。
4. **自有域名 = 资产**：预留接自定义域名的能力（Vercel 自定义域名）。
5. **实践挂靠**：项目页与博客互相印证——讲"是什么"的同时亮出"我真做了什么"。

### 成功标准（怎么算做好）
- 文章和项目以**公开、有日期、可检索**的形式沉淀下来。
- 用 AI 搜索我的名字 / 项目时，结果里能出现并正确引用这个站。
- 我每写完一篇文章，能**低摩擦**地让它出现在站上（理想接近一键发布）。

## Part C · 实现方案（已与我讨论确定）

### 拓扑：两个独立仓库 + git submodule
网站**单独立项**，与写作仓库是两个独立 git 仓库 / 两个独立 Cursor 工作区，互不污染（规则、索引、技能都干净）。网站把写作仓库作为**内容源 submodule** 挂进来。

```
网站仓库 (本文件夹)
├── content/writing/        ← git submodule，指向写作仓库
│   └── 文章/<YYYY-MM-DD-主题>/02-成稿/*.md
├── src/                     ← Astro 源码
├── .cursorignore            ← 把 content/writing/ 加进去（不影响构建，只避免污染 AI 索引）
└── ...
```

- 写作仓库（私有，内容源）：`https://github.com/hahaszd/insights-and-plan.git`
- submodule 挂载路径：`content/writing/`

### 技术栈
- **Astro + TypeScript**，静态生成（SSG）。
- 部署 **Vercel**。

### 内容源解析（关键：成稿没有 frontmatter）
文章在 `content/writing/文章/<YYYY-MM-DD-主题>/02-成稿/*.md`，这些 md **没有 YAML frontmatter**，需要写一个**自定义 Astro 内容加载器**（glob `content/writing/文章/**/02-成稿/*.md`）按约定推断：
- **标题**：取正文首行的 `# H1`
- **日期**：取自所在文件夹名 `YYYY-MM-DD-主题`
- **摘要**：取正文开头第一个 `> 引言` 区块
- **slug**：取文件夹名里「主题」那一段
- **多版本择一**：一个 `02-成稿/` 里可能有多个版本（如"聚焦版/发布版"）——优先文件名含「发布版」的；否则取唯一文件；都不满足就跳过并告警
- **可选 frontmatter 覆盖**：如果某篇成稿以后手动加了 frontmatter（`published / summary / slug / cover` 等），优先用 frontmatter，没有就回退到上面的推断。**不要强制要求我改写作流程。**

### 页面清单
- **首页 / 关于**：个人介绍，一句话定位，导航到博客与项目。
- **博客列表**：卡片含标题、日期、摘要。
- **博客详情**：渲染 Markdown 正文，含 metadata / OG。
- **项目展示页**：先用我**手填的数据文件**（如 `src/data/projects.ts` 或 `.json`）占位，字段按"做了什么/怎么做/结果/日期/GitHub 链接"。

### 发布流程（submodule 的额外一步，封装成脚本）
写完新文章后的流程：
1. 在**写作仓库**里 `git add/commit/push`（文章才会出现在 GitHub 上）。
2. 在**网站仓库**里跑 `npm run publish`：脚本内部 `git submodule update --remote` + `git add` + `commit` + `push`。
3. Vercel 自动部署。

请实现这个 `npm run publish` 脚本，让发布尽量接近一键。

### 部署（Vercel + 私有 submodule 授权）
- 写作仓库是**私有**的，Vercel 拉取私有 submodule 需要授权：
  - 用同一个 GitHub 账号（`hahaszd`）通过 Vercel 的 GitHub App 授权**两个仓库**的访问；或
  - 配置 deploy key。
- 部署后验证：那 3 篇已有文章能自动出现在站上。

## Part D · 建议执行步骤（给新窗口 agent）

1. **环境与脚手架**：在本文件夹用 `npm create astro@latest`（TypeScript、最小模板），初始化 git。
2. **建网站的 GitHub 仓库**：用已登录的 `gh`（账号 `hahaszd`）建一个仓库并 push（公开/私有由我定，建议公开，因为站点内容本就要给 AI 看；但仓库公开与否和站点公开与否是两回事，可问我）。
3. **挂 submodule**：`git submodule add https://github.com/hahaszd/insights-and-plan.git content/writing`，并把 `content/writing/` 写进 `.cursorignore`。
4. **自定义内容加载器**：实现 Part C 的成稿解析逻辑。
5. **页面**：首页/关于、博客列表、博客详情、项目页（手填数据占位）。
6. **样式**：干净克制、移动端适配。
7. **AI 可见性**：sitemap、RSS、OG/metadata，考虑 llms.txt。
8. **发布脚本**：`npm run publish`。
9. **部署 Vercel**：处理私有 submodule 授权，验证文章自动上线。
10. **双语预留**：路由/架构上为中英双语留好位置（本期可只做中文）。

> 过程中遇到需要我拍板的点（仓库公开与否、自定义域名、配色取向等），集中问我，不要逐步停下等我。
